# Cierre de Auditoría

Esta sección resume los criterios de aceptación, supuestos, restricciones y aprobaciones del Project Charter.

## Criterios de aceptación

La auditoría se considera completada cuando se cumplen condiciones como:

- ejecutar las fases definidas;
- aplicar el checklist SDLC;
- recopilar evidencia suficiente;
- revisar backend, frontend, Jira, pruebas y seguridad;
- elaborar matriz de hallazgos;
- elaborar matriz de riesgos;
- emitir informe preliminar e informe final;
- proponer plan de acción correctiva;
- archivar papeles de trabajo;
- cerrar formalmente la auditoría.

## Supuestos

Los supuestos son condiciones que se consideran válidas para planificar la auditoría:

- ModularSoft colaborará con evidencias;
- la documentación disponible será usada como fuente inicial;
- el sistema será evaluado en su estado actual;
- no se asumirán funcionalidades futuras como implementadas;
- las herramientas como Jira, GitHub, Jenkins, Swagger, k6, Playwright y Sonar aportan evidencia útil.

## Restricciones

Las restricciones establecen límites del trabajo auditor:

- tiempo limitado;
- recursos limitados;
- acceso no permanente al entorno funcional;
- revisión basada en evidencias disponibles;
- no modificación directa del código;
- no pentesting profundo;
- no certificación oficial;
- finalidad académica.

## Aprobaciones

El Project Charter incluye una sección de aprobación formal para el patrocinador, auditor líder, representante del equipo auditado, profesor responsable y equipo auditor.

!!! tip "Forma simple de explicarlo"
    El cierre indica cuándo la auditoría puede darse por terminada y bajo qué condiciones fue planificada.
